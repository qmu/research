# LLMの応答速度:持続スループット、TTFT、そして総レイテンシ

意思決定に最も明確に関わる発見は、**典型的な短〜中程度のやり取りでは、持続スループットではなくtime-to-first-token(TTFT)が総応答時間を支配する**ということであり、しかもTTFTはモデル、プロバイダー、さらには同一モデル内のeffortレベルによってすら、非常に大きく、しばしば予測不可能なほどにばらつく。GPT-5.4 nano(high effort)は、短文回答プローブにおいて467msのTTFT、165 tokens/secのスループット、596msの総レイテンシを記録した一方で、Grok Build 0.1は、生成が始まってからは227 tokens/secという十分なスループットを示したにもかかわらず、9,531msのTTFT(総レイテンシ9.68秒)を記録した。Gemini 3.1 Pro(low effort)とGPT-5.3 Codex(xhigh)も同様に、それぞれ6.5秒、2.4秒という数秒単位のTTFTを示し、生のトークン生成速度による優位性を完全に打ち消してしまっていた。これは、レイテンシに敏感なインタラクティブな用途(チャット、オートコンプリート、エージェント的ループなど)においては、TTFTこそが最適化すべき指標であり、しかもモデルの階層、プロバイダー、価格からは信頼できる形で予測できないことを示唆している。

判断とは切り離して考えるべき第二の観察点として、**「effort」や推論設定を高くすることが、常に一定方向で速度と品質をトレードオフするわけではない**という点が挙げられる。あるときはTTFTと総レイテンシの双方がおおよそ倍になり(Claude Opus 4.8のlow→xhigh:TTFT 1237ms→2143ms)、あるときは極端な外れ値を生み(Claude Sonnet 5のmax effortではTTFT 16,739ms、さらにそのxhigh設定はこの1回の試行において長さ指示の正答率が0%だった)、またあるときはむしろeffortを上げた方が速度が*向上する*(このサンプルでは、GPT-5.4 nanoのhigh effortは「none」effortよりも高速だった)。各構成につき試行数がn=1であることを踏まえると、これらのeffortレベル間のばらつきは、真にアーキテクチャ由来の挙動を反映している可能性もあれば、単に計測当日のノイズやキューイングによる変動である可能性もある——本成果物ではこの両者を区別できず、信頼区間を定めるための標準偏差や反復試行のデータも存在しない。

持続スループットに関しては、small/nano階層のモデルがフラッグシップモデルよりも高速であるケースが頻繁に見られた——GPT-5.4 nano(high effort)は165 tokens/secを、GPT-5.1 Codex mini(medium effort)は219 tokens/secを記録しており、どちらもClaude Opus 4.8(約60-66 tokens/sec)やGPT-5.5(約28-38 tokens/sec)といった多くのフロンティア級モデルの数値を大きく上回っている。これは真の意味での速度/コスト/性能のトレードオフであり、小型モデルは出力トークンあたりのコストも10-50倍安い(例:GPT-5.4 nanoは出力$0.60/MTokであるのに対し、GPT-5.5は$30/MTok)ため、一部のフラッグシップ構成に見られるような深い推論や大規模なスキーマ処理よりも、生のスループットとコストが重視される場面では魅力的な選択肢となる。注目すべきは、スキーマ処理能力(到達可能な最大の深さ/幅)も速度とは独立にプロバイダーごとに異なるという点であり——Grok 4.3とGrok 4.20はスキーマの深さ48に到達したのに対し、GPT-5.x系モデルは深さ10前後で頭打ちとなっている——したがって、純粋に速度だけを最適化しようとする購入者は、高速な小型モデルが自分たちの求める構造化出力の複雑さに実際に対応できるかを確認すべきである。

指示追従の信頼性(長さの正確さ)は、選択において重要な意味を持つ形でしばしば速度と逆相関するもう一つの軸である。Claude Sonnet 5はxhigh effortにおいて高速だった(TTFT 961ms、約90 tokens/sec)ものの、長さの正確さは0%であり、そのmax-effort構成では16.7秒のTTFTと0%の長さ正確さが組み合わさっており——このサンプルにおいて両次元で同時に最悪の結果となっていた。これは、速度指標だけではモデル選定に不十分であることを裏付けている。明示的な制約を無視する高速で安価な応答は、下流での修正コストという形で、遅くとも指示に従う応答より高くつく可能性がある。

最後に、これらの数値はスナップショットとして読むべきであり、統計的に頑健なベンチマークとして読むべきではない点に注意が必要である。**本成果物のあらゆるセルは、モデル/effort/プロバイダーの組み合わせごとに単一試行(n=1)を表しており**、報告されている標準偏差が0であるのは、単にばらつきを明らかにするための反復試行が行われなかったためにすぎない。実際のAPIレイテンシは、負荷、地域、時間帯によって変動することが知られており、ここに含まれるいくつかのデータ点はもっともらしい外れ値のように見える(例えば、o4-miniのhigh effortにおけるTTFT 7.4秒は、そのlow effortの930msと対照的である、あるいはGPT-5.3 Codexのxhighにおける不正な0 tokens/sec
