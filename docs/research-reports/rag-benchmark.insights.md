On this tiny SciFact-derived slice (3 queries, 5 documents, 1 relevant document per query), every backend that returned results achieved perfect retrieval quality: sqlite-vec, OpenAI's File Search, AWS S3 Vectors, and Cloudflare Vectorize all measured recall@3, nDCG@3, and MRR of 1.0. This means retrieval quality is not the differentiator here — with a query set this small (95% CI on recall spans as low as 0.44 given only 3 queries), quality comparisons are essentially uninformative, and the real decision signal is operational: latency, ingest cost, and architecture. Cloudflare AutoRAG is the exception — it produced a hard error, not a low score. Its dashboard-only R2 bucket-binding requirement meant the REST-driven ingest never populated a searchable index, so its 0.0 metrics reflect an unattainable measurement, not a quality failure.

The starkest trade-off is latency versus management overhead. Self-managed, fixed-embedding stores are dramatically faster at query time: sqlite-vec's p50 query latency was a fraction of a millisecond (~0.06ms) and S3 Vectors measured 0ms p50/p95, though the S3 Vectors run is explicitly marked "fixtured" rather than live-measured, so that zero should be read as a stand-in, not a real network round-trip. Cloudflare Vectorize, by contrast, is a genuinely measured self-managed store and lands at a p50 of ~128ms — still far below the managed OpenAI vector store, whose p50 query latency measured ~1,515ms (with a p95 near 1,795ms), roughly an order of magnitude slower. That gap reflects the cost of a fully managed, internally-embedded pipeline versus a store that only indexes vectors you already computed.

Ingest cost tells a complementary story. sqlite-vec's ingest was near-instant (~0.24ms), while Cloudflare Vectorize's measured ingest ran to ~28 seconds — largely because Vectorize's mutations are eventually consistent and the benchmark's polled propagation wait is folded into that number, not because writing vectors is inherently slow. OpenAI's managed store measured ~15.8 seconds to ingest, consistent with server-side chunking and embedding happening as part of the call. So the "managed" option trades slower, costlier ingestion and much higher per-query latency for zero embedding-pipeline maintenance; self-managed fixed-embedding stores are cheap and fast but require you to run and maintain your own embedding step.

On direct cost, only OpenAI's run carries a nonzero measured cost in this artifact — $0.0375 for the run, driven by its $2.50-per-1k-calls search pricing — while sqlite-vec, S3 Vectors, and Vectorize all measured $0 for this benchmark's volume (S3 Vectors and Vectorize note their production pricing schemes but confirm this workload sits under/at the free tier). This makes the managed OpenAI option the only one with a visible per-query cost signal at this scale, though it also offers built-in metadata filtering and no infrastructure to operate — a real trade worth weighing against its latency and ingest-time costs.

These results should be read as a narrow, low-n snapshot: 5 trials for operational metrics, only 3 queries underlying retrieval quality (hence wide confidence intervals despite perfect point estimates), and one backend (S3 Vectors) reporting fixtured rather than live operational numbers. The dataset is a small repository-authored fixture, not a full SciFact evaluation, so these findings characterize relative latency/cost/architecture behavior at toy scale — not statistically robust retrieval-quality rankings, and not a guarantee of behavior at production data volumes.
