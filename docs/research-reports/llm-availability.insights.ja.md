# API可用性

本レポートは、各プロバイダーの**公開ステータスページのインシデント履歴**をもとに構築された、各プロバイダーの可用性に関する**縦断的な記録**です。

## 1. 調査の目的

本調査の目的は、各プロバイダーが自社インシデントについて報告している内容を、比較可能かつ再現可能な形で記録し、その記録からローリング可用性指数を導出することである。モデルAPIそのものを検証するものではなく、参照元のステータスページに記載されている範囲を超えてプロバイダーの信頼性を主張するものでもない。

## 2. 測定対象

### 対象モデル

対象セットは、本リポジトリにおけるLLMプロバイダー群のプロバイダーステータスソースである。プロバイダーはアルファベット順に列挙され、記録は`2026-07-09T03:55:03.571Z`時点で評価されている。

### 対象メトリクス

メトリクスは、算出された30日間および90日間の稼働率指標、インシデント件数、重大/致命的インシデント件数、加重ダウンタイム分数、計画メンテナンス分数、90日間の日次トレンドポイント、および各ステータスソースの出典情報から構成される。

## 3. 範囲と制約

- **SLAでもランキングでもありません。** 導出された稼働率は、各プロバイダーが自ら報告した内容に基づく加重指数であり、サービス保証や「最も信頼性の高いプロバイダー」であることを主張するものではありません。
- 重み付け：重大障害（critical outage）×1.0、大規模障害（major）×0.5、軽微障害（minor）×0.1。計画メンテナンスは除外し、別途報告します。
- ステータスページはそれぞれ異なる粒度でインシデントを記録しているため、1件のインシデントが指数にカウントされるのは最大**24時間**までとしています。ただし、インシデント記録にはその全期間が保持されます。
- 出典：コミット済みの累積履歴（キー不要、決定論的レンダリング）。

## 5. 考察

これらの数値は、あくまでステータスページ集計から得られた観測結果として読み取るべきである。導出された稼働率の値が高いということは、集計元の記録において当該期間中の重み付きインシデント時間が小さかったことを意味するにすぎず、実際のAPI可用性や将来の信頼性を証明するものではない。

## 6. 再現方法

### 再現手順

```sh
git clone https://github.com/qmu/research
cd research/packages/tech
npm install

npm run availability:fixture
npm run availability:estimate
npm run availability
```

### 再現コスト（目安）

`availability:fixture` はキー不要かつ無償です。`availability:estimate` はLLM抽出ステップの料金を見積もるのみで、実際の取得・書き込みは行いません。`availability` は公開ステータスページを取得し、設定されたLLM抽出器を使用するため、プロバイダー・モデルごとのトークンコストが発生します。

### クリーンアップ

フィクスチャ経路では外部リソースは作成されません。実データ経路ではローカルの累積履歴とレポート成果物が更新されるため、コミット前にそれらのファイルを確認してください。プロバイダー側のリソースは作成されません。
