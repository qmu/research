# トークン計測

本レポートは、LLM APIが課金対象とする入力トークンを**プロバイダのトークナイザーライブラリなしに**カウントできるかどうかを測定する。プロバイダがトークナイザーの語彙とマージルールを公開している場合（OpenAIのencodings、オープンウェイトモデルの`tokenizer.json`）は、自前実装のbyte-pair-encodingカウンターが完全一致するかを検証する。トークナイザーが非公開の場合（Anthropic Claude、Google Gemini）は、プロバイダの非課金カウントエンドポイントに対してキャリブレーション済み推定器をフィットさせ、その誤差幅を報告する——明記された幅を伴う推定であり、誤った精度の主張は行わない。この2つのカウント方式の違いと、それらを取り巻く課金上のエッジケース（日本語テキスト、出力トークンの事前推定、キャッシュ／ツール課金、画像変換）が、プロバイダ非依存でプリンシパル単位の使用量メータリングを構築できるかどうかを決定づける。

## 1. 調査の目的

本調査の目的は、4つのプロバイダーファミリーについて、入力トークンがどのようにカウントされるか、どのような条件下でライブラリに依存しない自己カウントが可能か、固定サンプルセットにおいてAPIが報告するカウントに対してどの程度の精度に達するか、そしてそのカウントに対してどのような価格体系が適用されるか——使用量メタリングおよびコスト帰属レイヤーが構築される際の基盤となる特性——を記録することにある。

## 2. 測定対象

### 対象モデル

対象は、精選されたレジストリ（`packages/tech/src/token-metering/models.ts`）内の4つのプロバイダーファミリーであり、それぞれ代表モデルを1つずつ選定し、カウント方式はプロバイダーが公開している内容に従う：

- **Anthropic Claude**（`anthropic-claude`、`claude-sonnet-5`に対して測定）：現行モデル向けのトークナイザーは非公開（アーカイブされた@anthropic-ai/tokenizerはレガシーなClaude 2のみを対象とする）であるため、正確な自己カウントは存在せず、自己カウントは課金対象外のcount_tokensエンドポイントに対して較正された推定値である。出典：https://platform.claude.com/docs/en/build-with-claude/token-counting （2026-07-17検証）。
- **OpenAI GPT**（`openai-gpt`、`gpt-5.5`に対して測定）：OpenAIはそのエンコーディングの語彙をランク付けされたバイト列（o200k_base、約200kエントリ）として、事前トークン化パターンとともに公開しているが、count-tokensエンドポイントは存在しないため、正解データは最小限の課金対象completionから得られるusage.prompt_tokensである。出典：https://developers.openai.com/api/docs/pricing （2026-07-17検証）。
- **Google Gemini**（`google-gemini`、`gemini-3.1-pro-preview`に対して測定）：SentencePiece系統。GoogleはGemmaトークナイザーモデルは公開しているが、現行のGemini APIトークナイザーは公開していないため、自己カウントは課金対象外のcountTokensエンドポイントに対して較正された推定値である。出典：https://ai.google.dev/gemini-api/docs/tokens （2026-07-17検証）。
- **OSS / ローカル（Qwen2.5）**（`oss-qwen`、`@cf/qwen/qwen2.5-coder-32b-instruct`に対して測定）：オープンウェイトモデルにはtokenizer.json（語彙＋順序付きマージリスト＋事前トークナイザーパターン、Qwen2.5はApache-2.0）が同梱されているため、正確な自己カウントが存在する。正解データは、ホスト型サービングスタック（Cloudflare Workers AI）が報告するusage.prompt_tokensである。出典：https://huggingface.co/Qwen/Qwen2.5-Coder-32B-Instruct （2026-07-17検証）。

### 対象メトリクス

主要メトリクスは、自己カウントによる予測値がAPI報告のトークン数と比較した場合の、ホールドアウトにおける平均絶対誤差および最大絶対誤差（%）であり、値が小さいほど良好とされ、クラスごとに合意された目標値は|error| ≤ 5%である。これに加えて、クラスごとの符号付き誤差幅[min, max]を用いる。記述的指標としては、フィッティングされたラッパーのオーバーヘッド（リクエストあたりのトークン数）、クラスごとの文字あたりトークン発生率、およびプローブ支出（USD、参考値）を用いる。コスト算出に用いる価格は、基盤モデルカタログに公開されているMTokあたりの単価である。

## 3. 範囲と制約

- **入力トークンのみ。** 出力トークンは実行前にカウントすることができない（モデルがいつ停止するかを決定するため）。その事前推定のセマンティクスは第5節でエッジケースとして分析し、実行後の計上はレスポンスの usage フィールドを読み取る。
- **カウントであり、課金の照合ではない。** 本チェックはトークン数を比較するものであり、請求書レベルの照合（丸め、最低料金、ティア割引）は範囲外である。
- **ファミリーごとに代表モデルを1つ。** カウントは `claude-sonnet-5`、`gpt-5.5`、`gemini-3.1-pro-preview`、`@cf/qwen/qwen2.5-coder-32b-instruct` に対して検証されており、同一ファミリー内の他のモデルは異なるトークナイザーを使用している可能性があるため、キャリブレーションを再利用する前に再検証しなければならない。
- **サンプルマニフェストは固定されている**（tm-v1: 30サンプル、クラスごとに10サンプル、半分はキャリブレーション用、半分はホールドアウト用）。精度に関する主張はこれらのクラスと長さについて成り立つものであり、その他のコンテンツクラス（例：base64 blob、密なUnicodeアート）は未検証である。
- **ラッパーのオーバーヘッドはフィッティングされたものであり、文書化されたものではない。** プロバイダーはチャットテンプレートのトークンコストを公開していないため、アフィンモデルはそれをフィッティングされた定数として吸収しており、これはここで使用されている単一ユーザーメッセージ形式のリクエストを前提としている。
- キーレスのフィクスチャ経路は決定的であり、明らかに合成的である（バイト数ベースの語彙、固定レート）。実際のエラー数は、承認済みのコスト上限内でオーナーがトリガーした実実行からのみ得られる（まず `--estimate` を実行すること）。
- **時点情報：** 測定された挙動は `2026-07-17T03:02:34.699Z` 時点における各プロバイダーのトークナイザーおよびカウントエンドポイントを反映している。語彙および価格は各行の情報源および検証日時点のものである。

## 5. 考察

`measured`という来歴を持つファミリーは、ホールドアウト誤差で比較できる。exact-BPEファミリーは、公開された語彙と公開された事前トークン化パターンが課金カウントを再現できるかを検証するものであり、推定器ファミリーは、クラスごとの文字数／トークン比モデルがどこまで信頼できるかを検証するものである。最大誤差が目標帯域内に収まっていれば、事前呼び出しのコスト予測とプリンシパルごとの帰属は自己カウントのみで実行できる。帯域外のクラスは、レスポンスのusageフィールドから事後的に計測する必要があり、その帯域が予測リスクを示している。

#### エッジケース1 — 日本語テキスト

日本語は英語の文字あたりトークン率の数倍でトークン化される（クラスごとの文字数／トークン比は7章参照）。語彙はラテン文字中心であるため、多くの漢字はバイト単位のピースにフォールバックする。英語の比率を前提としたコスト予測は日本語入力を過小評価する。キャリブレーションにおけるクラスごとの比率がその補正であり、日本語のホールドアウト帯域がその信頼度を示している。

#### エッジケース2 — 出力トークンは事前カウントできない

2つの異なる量を混同してはならない。**実行前の見積もり**（モデルがいつ停止するかを決めるため、境界値しか存在しない — リクエストのmax-tokens上限がハード上限であり、ワークロードごとの過去の出力／入力比が期待値を与える）と、**実行後の記録**（レスポンスのusageフィールドから得られる正確な値 — 出力トークン、およびプロバイダが出力として課金する場合のreasoningトークン）である。メータリング層は実行後の記録を保存し、実行前の境界値はクォータ許可のためだけに使用し、課金には決して用いない。

#### エッジケース3 — キャッシュとツール利用の課金

キャッシュされたリクエストやツールを伴うリクエストは、同じトークン数を異なる料率で課金するため、メーターは単一の数値ではなく内訳を保持しなければならない。プロンプトキャッシュの書き込みは基本入力料率より高く課金され、キャッシュの読み取りはそのごく一部の料率で課金される（各プロバイダは独自の倍率とキャッシュ可能な最小長を公開している）。レスポンスのusageフィールドはキャッシュ書き込みトークンとキャッシュ読み取りトークンを別々に報告し、メーターは各バケットをそれぞれの料率で価格付けしなければならない。ツール定義はプロンプトにシリアライズされ、通常の入力トークンとして課金される — Anthropicのcountエンドポイントにおける1つのツール定義の実測オーバーヘッドは7章にある — また、プロバイダ側のツール結果（例：ウェブ検索）はコンテキストに戻り、次のターンで入力として課金される。

#### エッジケース4 — 画像とマルチモーダル入力

画像のトークン変換はBPEではなく数式ベースである。Anthropicは幅×高さ/750（自動ダウンスケーリングによる上限あり）を文書化しており、Googleは384px以下の画像に対して一律258トークン、それを超える場合は768×768タイルごとに258トークンを文書化しており、OpenAIはモデルファミリーごとにベース＋タイル方式のスケジュールを文書化している。固定した1枚のPNGに対するプローブ結果は7章にある。メーターはこれらの変換をプロバイダごとに実装すべきであり、テキストカウントと同じインターフェースの背後に置き、数式のバージョンを記録すべきである — プロバイダはモデル世代間で変換スケジュールを変更するためである。

#### 実装方針 — プリンシパルごとの利用量メータリング（設計章）

本調査を利用する設計（plggトークンメータリングライブラリおよびその利用者、例えばqmu-co-jp sync-controller）は、呼び出し元が保持する利用記録を通じて、利用量を**プリンシパル**（qmu.appプランのRBAC/PBACサブジェクト）に帰属させる：

- **レコード形式**：`{ recordedAt, principalId, model, inputTokens, cachedInputTokens, outputTokens, costUsd, countingMethod, calibrationVersion }` — 利用内訳はプロバイダのusageフィールドを反映し、コストは入力／キャッシュ／出力の内訳を保持し、カウント方式とキャリブレーションのバージョンにより、保存されたすべての数値が再導出可能になる。
- **データ最小化／主権**：レコードはカウントと金額のみを保存し、プロンプトや補完テキストは保存しない。`principalId`は不透明な内部識別子である（観測性ポリシーの構造化ログ制約に従い、個人識別情報ではない）。削除のセマンティクスはスキーマ設計時に決定される — 利用記録は課金の証跡であるため、アカウント削除は`principalId`を匿名化する（人物を不可逆的に切り離す）のであって、財務上の合計を破棄するのではない。この保持方針は開示されている。
- **メトリクス出力**：集計値は観測性ポリシーに従い、プロバイダ固有のダッシュボードではなく、ベンダー中立な形式（プリンシパル／モデル／トークンバケットごとのOpenMetricsカウンター）でエクスポートされる。
- **ブランド型**：トークン数とUSD金額は別々のブランド付き数値型であり、価格を通じた明示的な変換なしにカウントが金額に加算されることはない — 本調査自体の検証ハーネス型が強制するコンパイル時の誤用防止と同様である。
- **ACL境界**：トークナイザーの語彙、countエンドポイント、usageフィールドは、腐敗防止層の背後にあるプロバイダ固有の詳細である。ドメインが目にするのは`countTokens(model, text)`と`estimateCost(model, usage)`のみである。

#### 依存関係の決定 — トークナイザーライブラリ（4点ログ）

1. **理由**：カウントにはトークナイザーが必要である。候補は`tiktoken`（OpenAIエンコーディング）、アーカイブ済みの`@anthropic-ai/tokenizer`（レガシーなClaude 2専用 — 現行モデルには使用不可）、Hugging Face `tokenizers`／transformers（OSSモデル用）である。
2. **評価**：いずれも信頼できるものであり、許可的なライセンス（MIT/Apache-2.0）である。しかし、それぞれが1つのプロバイダのみをカバーしており、Anthropicの現行モデルをカバーするものは存在せず、これらを採用すると、プロバイダごとのネイティブ／WASM依存関係がすべての利用者に持ち込まれることになる。
3. **決定と監視**：ベンダー中立性ポリシーの「デフォルトで実装する」原則に従い、自前実装を採用する。BPE推論ループは*公開データ*（語彙）に対して約150行であり、本トピックの繰り返し試行そのものが監視計画となる — 実行のたびに実際のAPIに対してカウントを再検証するため、トークナイザーの変更はホールドアウト誤差の回帰として表面化する。これは、ライブラリ採用者がリリースノートで待つのと同じシグナルである。
4. **撤退戦略**：カウントインターフェース（`countTokens`）が実装を隠蔽している。もしプロバイダが、自己カウントでは帯域内に再現できないような公開ルールを持つエンコーディングを提供した場合、そのプロバイダに限り、同じインターフェースの背後にリファレンスライブラリを採用することができ、`docs/dependency-decisions.md`に記録される。

## 6. 再現方法

### 再現手順

```sh
git clone https://github.com/qmu/research
cd research/packages/tech
npm install

# キー不要のセルフテスト（決定論的な固定語彙とカウント）:
npm run research -- token-metering --fixture

# コストのプレビュー、その後オーナー限定の実実行（プロバイダーキーが必要）:
npm run research -- token-metering --estimate
npm run research -- token-metering --real
```

実実行では、公開されている語彙（o200k_base.tiktoken; Qwen2.5 tokenizer.json）を `packages/tech/.cache/token-metering/` に取得する — これらは `docs/dependency-decisions.md` に記録されたデータダウンロードであり、コミットされることはない。

### 再現コスト（目安）

フィクスチャ経路はキー不要かつコストゼロである。実トライアルでは、課金対象外の2つのカウントエンドポイント（Anthropic count_tokens、Gemini countTokens — $0）を読み取り、2つの使用量プローブ系統（`gpt-5.5` と Workers AI の Qwen モデルそれぞれに対して30件の短いプロンプト）に対してサンプルごとに最小限の課金対象コンプリーションを1回実行する: 見積もり経路では、これを1トライアルあたり数セント程度と算出する。ミッション全体の測定に合意された上限は$5であり、`--estimate` は最初に実行しなければならない。

### クリーンアップ

実実行では何もプロビジョニングしない（ステートレスなAPI読み取りと最小限のコンプリーション）ため、後始末すべきものはない。ローカルのMarkdown/JSON成果物と語彙キャッシュを `packages/tech/.cache/` 配下に書き込む — コミット前に成果物をレビューすること。キャッシュはgitignore対象である。
