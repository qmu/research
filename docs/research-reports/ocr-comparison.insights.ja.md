# OCR能力

本レポートは、LLM基盤調査のためのOCR比較ハーネスを記録するものです。コミットされたレポートは、キー不要のフィクスチャパスによって生成されています。

## 1. 調査の目的

決定論的なドキュメント画像に対してOCRを実行し、書き起こしテキストと構造化フィールドの両方を返すことができる基盤モデル統合はどれかを検証することが目的である。

## 2. 測定対象

### 対象モデル

対象行は、OCRランナー内のモデル候補である。実際のモデル数値は、行の provenance が `measured` である場合にのみ存在する。`out-of-scope` 行は、登録済みの `VisionClient` アダプタを持たないモデルを示す。

### 対象メトリクス

測定対象のメトリクスは、文字誤り率（CER）、単語誤り率（WER）、およびフィールドレベルの構造化抽出精度である。必須フィールド: `document_type`、`vendor`、`document_id`、`date`、`currency`、`total`。

## 3. 範囲と制約

- フィクスチャの行はハーネスチェックのみを対象とする。これは、文書レンダリング、ビジョンポートの結線、CER/WER、フィールドスコアリング、集計、レポートレンダリングが決定的であることを証明するものである。
- `out-of-scope` と記された行は、画像パスを通じて送信されない。プロバイダ／モデルは、OCR実行に参加する前に明示的な `VisionClient` アダプタを必要とする。
- 実際のモデル数値は、オーナーが `--estimate` を確認したうえで実経路を実行するまで、未計測のままとなる。
- 合成フィクスチャは、クリーンなLatin／数字の文書レイアウトのみをカバーする。日本語OCR、手書き、傾き、スタンプ、スキャン、写真、複数ページ分割は測定対象外である。

## 5. 考察

`measured` の由来を持つ行はOCR品質について比較可能である。`out-of-scope` の由来を持つ行は、画像パスを一度も通過していないため、OCRの失敗ではなく統合カバレッジの欠落である。

## 6. 再現方法

### 再現手順

```sh
cd packages/tech
npm install

npm run ocr:fixture
npm run ocr:estimate
npm run ocr:real
```

### 再現コスト（目安）

`ocr:fixture` はキー不要でコストもかかりません。`ocr:estimate` はオーナー限定の実処理パスをプレビューします。`ocr:real` は設定済みのビジョン対応プロバイダーを呼び出し、プロバイダー側のトークン/画像課金が発生します。

### クリーンアップ

OCRのフィクスチャおよび実処理パスは、外部のクラウドリソースを作成しません。これらはローカルのレポート/データ成果物を生成するため、コミット前にそれらのファイルを確認してください。

**採点。** CERとWERは編集距離を参照文字列の長さで割った値です。フィールド精度は、正規化されたフィールドの完全一致数を必須の正解フィールド数で割った値です。正規化：採点前に連続する空白を1つのASCIIスペースに圧縮し、両端をトリムし、改行コードを正規化します。CER／WERでは句読点を保持します。フィールド値では数値の合計からグルーピング用のカンマを除去し、通貨コードは大文字に統一します。
