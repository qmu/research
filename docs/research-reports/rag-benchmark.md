# RAG backend benchmark

This report records a benchmark harness for vector-store and RAG-database backends. It separates curated backend facts from measured retrieval and operational behavior. The committed report is generated by the keyless fixture path; real local sqlite-vec runs use the same dataset, embedding, scoring, and report renderer.

## Methodology

**Dataset.** The benchmark uses SciFact miniature subset (`scifact-mini`), a committed miniature subset shaped after SciFact. Source: https://github.com/allenai/scifact. License note: SciFact is Apache-2.0; this miniature fixture is repository-authored from SciFact-style claims. The fixture contains 5 documents, 3 queries, and 3 relevance judgments.

**Metrics.** Retrieval quality is recall@3, nDCG@3, and MRR against the committed qrels. Operational measurements are ingest time, query latency p50/p95, measured scale, and estimated cost.

**Embedding boundary.** Self-managed stores use one fixed local embedding model (`fixed-hash-embedding-v1`) so the store comparison is isolated from embedding-provider behavior. Managed stores added by later tickets must mark `isolatedStore: false`.

## Data transparency

The complete run record is committed as [`rag-benchmark.data.json`](./rag-benchmark.data.json). It contains the dataset rows, backend registry facts, query-level results, retrieval metrics, operational timings, provenance, and generated timestamp.

## Scope and limitations

- This foundation ticket includes sqlite-vec plus a deterministic fixture path. Cloud backends are separate follow-up tickets.
- The committed fixture report is a CI self-test. A real sqlite-vec run is local and keyless, but its timings depend on the machine.
- The miniature dataset is intended to verify the harness and metric calculations; it is not a statistical claim about backend quality at production scale.
