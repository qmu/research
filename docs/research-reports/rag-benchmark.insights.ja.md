# ベクトルDBの比較

本レポートは、ベクトルストアおよびRAGデータベースバックエンドを対象としたベンチマークハーネスの記録である。整備済みのバックエンド情報と、実測された検索・運用挙動とを分離して扱う。

## 1. 調査の目的

埋め込み境界を明確に保ちながら、RAGバックエンド間の検索品質と運用挙動を比較することを目的とする。

## 2. 測定対象

### 対象モデル

対象セットは5個のバックエンド構成です。セルフマネージド型のストアは固定のローカル埋め込みモデル（`fixed-hash-embedding-v1`）を1つ使用し、フルマネージド型の行はマネージドスタック全体を測定対象とします。

### 対象メトリクス

検索品質は、コミットされたqrelsに対するrecall@3、nDCG@3、MRRであり、3クエリにわたる値 ± 95%信頼区間として示します。運用面の測定項目は取り込み時間とクエリレイテンシp50/p95であり、5回の試行にわたる平均 ± 標本標準偏差として示し、加えて実測されたスケールと推定コストを示します。

## 3. 範囲と制約

- コミットされたレポートは、CIのセルフテストとしてキーレス・フィクスチャ経路によって生成されます。実際の実行では同一のデータセット、スコアリング、レンダラーを使用しますが、タイミングはマシンとネットワークに依存します。
- フルマネージドのバックエンドはスタック全体を測定します。認証情報が存在しない場合の実行では、そのカードは`fixtured`としてレンダリングされ、決して偽装されることはありません。
- この小規模なデータセットは、ハーネスとメトリクス計算を検証することを目的としており、本番規模におけるバックエンド品質についての統計的な主張ではありません。

## 5. 考察

- **sqlite-vec** — ストア分離型（固定embedding）。この実行における検索品質は決定論的であり、95%区間は3件のクエリサンプルに対するものであって、試行間のばらつきではない。運用メトリクスは5回の試行における平均値 ± 標本標準偏差である。
- **OpenAI vector store (File Search)** — ホールスタック型であり、ストア分離型ではない（マネージドembedding）。File Searchはドキュメントをサーバー側でチャンク化し（デフォルトで800トークンのチャンク、400トークンのオーバーラップ）、内部でembeddingを行う。インデックス化される単位はOpenAI側のチャンクであり、コミットされたドキュメントそのものではない。この実行における検索品質は決定論的であり、95%区間は3件のクエリサンプルに対するものであって、試行間のばらつきではない。運用メトリクスは5回の試行における平均値 ± 標本標準偏差である。
- **AWS S3 Vectors** — ストア分離型（固定embedding）。我々が提供する固定embeddingベクトル（float32、cosine）を格納する、自己管理・ストア分離型の読み取りである。AWS認証情報と、S3 Vectorsが利用可能なリージョン（ap-northeast-1で検証済み）が必要である。IAM: s3vectors:CreateVectorBucket/CreateIndex/PutVectors/QueryVectors/DeleteIndex/DeleteVectorBucket。この実行における検索品質は決定論的であり、95%区間は3件のクエリサンプルに対するものであって、試行間のばらつきではない。運用メトリクスは5回の試行における平均値 ± 標本標準偏差である。
- **Cloudflare Vectorize** — ストア分離型（固定embedding）。我々が提供する固定embeddingベクトル（v2インデックス、cosine、次元数は[32,1536]の範囲）を格納する、自己管理・ストア分離型の読み取りである。書き込みは結果整合性（およそ5-15秒）であるため、測定されたingestにはポーリングによる反映待ちの時間が含まれている。認証: CLOUDFLARE_ACCOUNT_ID、およびVectorize編集権限を持つAPIトークン。この実行における検索品質は決定論的であり、95%区間は3件のクエリサンプルに対するものであって、試行間のばらつきではない。運用メトリクスは5回の試行における平均値 ± 標本標準偏差である。
- **Cloudflare AutoRAG** — ホールスタック型であり、ストア分離型ではない（マネージドembedding）。完全マネージド型であり、R2バケットから取り込みを行い、内部でembedding・インデックス化を行う（ホールスタック型であり、ストア分離型ではない）。インデックス化は非同期であり（6時間サイクル、またはレート制限付きの強制同期）、そのR2データソースバインディングはダッシュボード経由でプロビジョニングされるため、REST APIのみによるライブのエンドツーエンド実行では、`measured`ではなく正直に`error`となる場合がある。この実行における検索品質は決定論的であり、95%区間は3件のクエリサンプルに対するものであって、試行間のばらつきではない。運用メトリクスは5回の試行における平均値 ± 標本標準偏差である。

## 6. 再現方法

### 再現手順

```sh
cd packages/tech
npm install

npm run rag:fixture
npm run rag
npm run rag:estimate
```

### 再現コスト（目安）

`rag:fixture` はキー不要かつコストなしで実行できる。`rag:estimate` は認証情報／プロバイダー経路のプレビューを行う。`npm run rag` はローカルの計算リソースおよび設定済みのマネージドバックエンドの認証情報を使用する場合がある。

### クリーンアップ

ローカルフィクスチャ経路では外部リソースは作成されない。マネージドバックエンドでの実運用実行ではプロバイダー側リソースのクリーンアップが必須であり、RAGランナーには後片付け処理と、孤立リソース確認用の `npm run rag:sweep` が含まれている。
