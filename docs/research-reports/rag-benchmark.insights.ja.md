# ベクトルDBの比較

本レポートは、ベクトルストアおよびRAGデータベースバックエンドを対象としたベンチマークハーネスを記録したものである。精選されたバックエンド情報と、実測されたリトリーバル（検索）性能および運用上の挙動とを分離して扱う。

## 1. 調査の目的

埋め込み境界を明確に保ちながら、RAGバックエンド間で検索品質と運用上の挙動を比較することを目的とする。

## 2. 測定対象

### 対象モデル

対象セットは5種類のバックエンド構成である。自己管理型ストアは固定のローカル埋め込みモデル（`fixed-hash-embedding-v1`）を1つ使用し、一方フルマネージド行はマネージドスタック全体を測定する。

### 対象メトリクス

検索品質は、コミット済みのqrelsに対するrecall@3、nDCG@3、MRRであり、3件のクエリにわたる値 ± 95%信頼区間として示される。運用面の測定項目は取り込み時間およびクエリレイテンシp50/p95であり、5回の試行にわたる平均 ± 標本標準偏差として示され、これに加えて測定されたスケールと推定コストが含まれる。

## 5. 考察

- **sqlite-vec** — ストア分離型（固定埋め込み）。今回の実行では検索品質は決定論的であり、95%区間は試行分散ではなく3クエリのサンプルに対するものです。運用メトリクスは5回の試行にわたる平均値±標本標準偏差です。
- **OpenAI vector store（File Search）** — フルスタック型であり、ストア分離ではない（マネージド埋め込み）。File Searchはドキュメントをサーバー側でチャンク分割し（デフォルトで800トークンのチャンク、400トークンのオーバーラップ）、内部で埋め込みを生成します。インデックス化される単位はコミットされたドキュメントではなく、OpenAIのチャンクです。今回の実行では検索品質は決定論的であり、95%区間は試行分散ではなく3クエリのサンプルに対するものです。運用メトリクスは5回の試行にわたる平均値±標本標準偏差です。
- **AWS S3 Vectors** — ストア分離型（固定埋め込み）。こちらが提供する固定埋め込みベクトル（float32、コサイン類似度）を保存する、自己管理・ストア分離型の読み取りです。AWS認証情報と、S3 Vectorsが利用可能なリージョン（ap-northeast-1で検証済み）が必要です。IAM: s3vectors:CreateVectorBucket/CreateIndex/PutVectors/QueryVectors/DeleteIndex/DeleteVectorBucket。今回の実行では検索品質は決定論的であり、95%区間は試行分散ではなく3クエリのサンプルに対するものです。運用メトリクスは5回の試行にわたる平均値±標本標準偏差です。
- **Cloudflare Vectorize** — ストア分離型（固定埋め込み）。こちらが提供する固定埋め込みベクトル（v2インデックス、コサイン類似度、次元数は[32,1536]の範囲）を保存する、自己管理・ストア分離型の読み取りです。更新は結果整合性（約5-15秒）であるため、ポーリングによる伝播待機は計測されたingestに含まれます。認証: CLOUDFLARE_ACCOUNT_ID と Vectorize編集権限を持つAPIトークン。今回の実行では検索品質は決定論的であり、95%区間は試行分散ではなく3クエリのサンプルに対するものです。運用メトリクスは5回の試行にわたる平均値±標本標準偏差です。
- **Cloudflare AutoRAG** — フルスタック型であり、ストア分離ではない（マネージド埋め込み）。完全マネージド型で、R2バケットから取り込み、内部で埋め込み・インデックス化を行います（フルスタック型であり、ストア分離ではない）。インデックス化は非同期で行われ（6時間サイクル、またはレート制限付きの強制同期）、そのR2データソースのバインディングはダッシュボード経由でプロビジョニングされるため、REST APIのみによるライブのエンドツーエンド実行では、`measured` ではなく正直に `error` となる場合があります。今回の実行では検索品質は決定論的であり、95%区間は試行分散ではなく3クエリのサンプルに対するものです。運用メトリクスは5回の試行にわたる平均値±標本標準偏差です。

## 6. 再現方法

### 再現手順

```sh
cd packages/tech
npm install

npm run rag:fixture
npm run rag
npm run rag:estimate
```

### 再現コスト（目安）

`rag:fixture` はキー不要でコストもかからない。`rag:estimate` は認証情報を要するプロバイダー経路を事前に確認するものである。`npm run rag` はローカルの計算資源、および設定済みのマネージドバックエンドの認証情報を使用する場合がある。

### クリーンアップ

ローカルのフィクスチャ経路は外部リソースを作成しない。マネージドバックエンドでの実運用実行では、プロバイダーのリソースをクリーンアップする必要があり、RAGランナーには後始末処理と、孤立リソースを確認するための `npm run rag:sweep` が含まれている。
