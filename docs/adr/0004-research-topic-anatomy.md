# 0004 — Research-topic anatomy

## Context

Two `packages/tech` research topics have now shipped through the same shape — the
seed `llm-benchmark` exact-match study and the `llm-model-comparison` study. The
second topic is a deliberately different shape (a multi-provider comparison mixing
curated catalog data with live measurement) yet reused the first's layering
without perturbing it. That repetition is evidence of a stable, copyable anatomy
worth recording so future topics are mechanical rather than improvised.
ADR 0002 already fixes the one-npm-project-per-group rule and the layered `src/`;
this ADR records the *internal* anatomy a single topic follows.

## Decision

A research topic is built from these parts:

- **Pure `domain/`** — types and logic (prompt builders, graders, scorers,
  report renderer) with no SDK, filesystem, or environment access. Every public
  domain function has co-located vitest boundary tests that run in CI as
  regression.
- **`vendors/<provider>/` anti-corruption layer** — each external SDK sits behind
  a domain-named contract (`LlmClient`, `CompletionClient`). Provider types,
  including raw usage shapes, stop at this layer; the normalization of each
  provider's differently-named fields into the domain shape is a pure, unit-tested
  helper. New dependencies are recorded in `docs/dependency-decisions.md`.
- **Thin `entrypoints/run-<topic>.ts`** — the only impure module. It owns
  orchestration constants (probe ladders, targets), argv, env, the clock, and file
  writes; it wires the registry/dataset through the domain logic and writes the
  result page. No correctness logic lives here.
- **Curated-registry-vs-live-measurement split** (where a topic has one) — static
  cited reference data is a separate type from observed measurement, with a
  required provenance flag so a non-measurement can never be rendered as a live
  one.
- **A keyless fixture path** — a deterministic fixture client lets the full
  pipeline and report regenerate with no credentials, so CI exercises
  dataset → grade → render end to end. CI runs only the fixture target.
- **`docs/research-reports/<slug>.md`** — the objective, reproducible result page,
  generated by the renderer, carrying the `description` frontmatter the corporate
  site requires and a Reproduce section.

## Alternatives considered

- **Per-topic ad-hoc structure**: let each topic choose its own layout. Rejected —
  it loses the cross-topic consistency that makes a new topic cheap to add and
  keeps the domain logic testable in isolation.
- **One wide vendor contract**: widen the seed's `LlmClient` to carry tokens and
  timing for every topic. Rejected — the comparison topic added a second narrow
  contract (`CompletionClient`) alongside the untouched `LlmClient`, so each topic
  depends only on the contract it needs and the seed's tests keep passing.

## Consequences

- Adding a topic is a known sequence: scaffold `domain/`, write pure tested logic,
  reach providers through an ACL, add a thin runner and a fixture path, render an
  objective report. `packages/tech/TEMPLATE.md` is the step-by-step recipe; this
  ADR records the anatomy the recipe produces.
- The honesty of a mixed curated/measured report is enforced structurally (a
  required provenance flag and renderer masking), not by discipline.
- The fixture path keeps CI credential-free and the result pages reproducible from
  a clone.
