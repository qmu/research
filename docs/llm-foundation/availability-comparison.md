## 5. 考察

2026-07-09時点で、このデータセットは各プロバイダー自身のステータスページにおけるインシデント履歴を影響度で重み付けし、直近30日間および90日間のウィンドウで集計したものである——これは独立した第三者による稼働率測定ではなく、SLA数値として読むべきものではない。最も判断材料として重要な点は、xAIのステータスページ(status.x.ai)が取得時にHTTP 403 Forbiddenを返し、インシデントデータが一切抽出できなかったことである。そのため同社のウィンドウはダウンタイム0、導出稼働率100%と表示されているが、これはサービスが完全に利用可能であると検証された結果ではなく、単にインシデントが0件しか取得できなかったために生じたものにすぎない。読者はxAIの数値を「クリーンなステータス履歴の証拠」としてではなく、「データ取得上の欠落」として扱うべきである。

データが取得できた3社のプロバイダーのうち、Anthropicの30日間の導出稼働率(94.73%)は、同社自身の90日間の数値(98.14%)を明らかに下回っている。日別の内訳を見ると、このギャップの大部分は2026-06-13の1日に集中しており、その日には1,390.2分のダウンタイムが報告されている——これは自己申告かつ影響度加重による一点集中であり、月全体に均等に広がる慢性的なパターンではない。OpenAIの30日間(98.19%)および90日間(99.35%)の導出稼働率は、いずれもAnthropicの対応する期間の数値より高く、インシデント件数も30日間で19件、90日間で25件と、Anthropicの40件・49件と比べて少ない——Anthropicのステータスページは同じ期間でおよそ2倍のインシデント件数を記録していたことになる。ただし、ここでいう「インシデント」とは各プロバイダーが自社ページ上でどのようにイベントを記録・分類するかに依存しており、件数だけをもって各社間の重大度を直接比較することはできない。Google Cloudのウィンドウは30日間で導出稼働率100%、90日間で99.89%を示しており、各ウィンドウで記録されたインシデントはそれぞれ1件のみ、全履歴を通じた総インシデント数は3件と、データを取得できた3社の中で最もインシデント件数が少なかった。

これらの数値が浮き彫りにする重要なトレードオフは、インシデント件数の細かさと見出しとなる稼働率パーセンテージとの関係である。あるプロバイダーは多数のインシデントを記録していても(Anthropicの40〜49件)、そのほとんどが短時間・低影響であれば依然として90%台後半の稼働率に収まり得る。一方で、Anthropicの2026-06-13の障害のような単一の大規模イベントは、90日間のトレンドがはるかに良好であっても、30日間の導出数値を大きく左右し得る。つまり、同じプロバイダーであっても30日間と90日間のウィンドウは異なる物語を語り得るのであり、「直近の」信頼性を評価する読者は、どちらか一方だけでなく両方のウィンドウを確認すべきである——短期ウィンドウでは下降トレンドを示すプロバイダー(Anthropic)であっても、90日間の実績としてはより良好な結果を示している場合がある。

最後に、全体を通していくつかの限界が存在する。これらの導出値の背後にある試行回数は明示されておらず、この手法は各プロバイダー自身のステータスページの公開慣行、インシデント分類、影響度の重み付けに全面的に依存している——独立した検証レイヤーは存在しない。この成果物は、単一時点(2026-07-09T03:55:03.571Z)で生成された固定データとして明示的に注記されており、継続的なライブフィードではない。また、xAIのデータ欠落は、ソースの取得可能性自体がプロバイダーごとに異なり、それがフラグ付けされなければ、誤解を招くほど「クリーン」に見える結果を静かに生み出し得ることを示している。これらの数値はいずれも、プロバイダーが公表するSLAコミットメントと同一視したり、その代替としたりすべきではなく、このデータのみから「より信頼性が高い」プロバイダーの順位付けを導き出すべきでもない。

## 6. 再現方法

調査は公開リポジトリ `qmu/research` で再生成できます。キー不要の `--fixture` は累積履歴から 30/90 日トレンドを描画し、real 経路は公開ステータスページを取得して LLM 抽出で履歴を更新します。

```sh
git clone https://github.com/qmu/research
cd research/packages/tech
npm install

npm run availability:fixture   # keyless: 累積履歴から描画
npm run availability:estimate  # 見積のみ
npm run availability           # real: 取得 + LLM 抽出で履歴更新
```

公開判断に使う場合は、取得元・as of・抽出モデル・provenance を記録し、各社の報告からの導出値であることを明記します。
