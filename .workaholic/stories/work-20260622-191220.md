## 3. Why

The repository must let a reader on qmu.co.jp clone it and reproduce a study, and
publish finished result pages to the corporate site. It is built to the workaholic
engineering policies: role-divided layout, one canonical `make` runner that CI
invokes, conservative vendor dependence with a decision log and anti-corruption
layers, pinned non-root containers, objective docs and ADRs, and a preview site
reachable by humans and AI agents at WCAG 2.2 AA.

The comparison instrument exists to answer a recurring question honestly: how do
current frontier-through-small models across four providers actually differ on
generation speed, latency, structured-output ceiling, and instruction-following —
measured live, over multiple trials, with every curated fact cited and every
non-measured cell flagged, never faked.

## 7. Successful development patterns

- **Keyless deterministic fixture as the CI self-test.** A byte-stable golden
  snapshot exercises the whole pipeline (including a stubbed judge) with no key or
  cost; real sweeps run only on demand, behind a printed cost estimate.
- **Live-probe-corrects-the-registry.** A real `--models` run plus a direct
  `GET /v1/models` and a with/without-parameter probe distinguishes a dead model id
  from a rejected parameter, so the registry is fixed from ground truth — dropped to
  `n/a`, never faked.
- **Scratch `OUTPUT_PATH` protects the committed fixture.** Real measurements write
  to a scratch path, so live verification never disturbs the byte-stable committed
  self-test the CI gate depends on.
- **Adaptive per-axis schema probe finds the model's real ceiling**, not the probe's
  — a fixed ladder only measured its own limit; a cap-cleared reading is reported as
  `">= cap"`.
