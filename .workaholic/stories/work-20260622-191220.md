## 3. Why

The repository must let a reader on qmu.co.jp clone it and reproduce a benchmark
or study, and must publish finished result pages to the corporate site as
foundational research. It is built to the workaholic engineering policies:
role-divided layout (`directory-structure`), one canonical `make` runner that CI
invokes (`command-scripts` + `ci-cd`), conservative vendor dependence with a
decision log and anti-corruption layers (`vendor-neutrality`), pinned non-root
containers (`containerization`), objective docs and ADRs (`objective-documentation`),
and a preview site reachable by humans and AI agents at WCAG 2.2 AA
(`accessibility-first`).

## 4. Outcome

The full reproducibility loop is verified end to end:

- `make install` / `make build` / `make test` / `make lint` all pass (industry +
  tech packages; 11 unit tests in the benchmark domain).
- `npm run benchmark:fixture` runs with no API key and writes
  `docs/research-reports/llm-benchmark.md`.
- The VitePress site builds and includes that page (sitemap generated).
- `make publish --dry-run` resolves the page to
  `../qmu-co-jp/docs/research/llm-benchmark.md` (Astro `/research` section),
  without writing to the sibling repo.

Decisions taken (recorded as ADRs): no workspaces / independent npm projects
(0001), one npm project per group with topics as subfolders (0002), VitePress
preview + Astro publish boundary as a Markdown file copy (0003).

## 6. Concerns

- **Publish is not yet an automated CI deploy stage** (raised and deferred in
  ticket #3). Wiring `make publish` into CI requires a qmu-co-jp deploy
  key/token for the cross-repo push. Today the reproducible logic lives in
  `scripts/publish-research.sh` and is documented in `CLAUDE.md` `## Deploy`;
  publishing is run by a maintainer. Not a blocker for landing the repo.
- **The committed `llm-benchmark.md` shows the fixture model at 100%.** This is
  framed in the page body as the pipeline self-test, not a real model result.
  Real numbers come from running with `ANTHROPIC_API_KEY`. Acceptable for a seed;
  a future run should publish a real-model comparison.
- **Accessibility tooling is an `optionalDependency`** so `make install` never
  breaks on platforms without Chrome (e.g. arm64). The `make a11y` check
  therefore runs only where Chrome is available (CI x64). Intentional trade-off.

## 7. Successful development patterns

- Verifying each ticket green before commit (`make` targets), and proving the
  cross-repo publish with a `--dry-run` against a temp directory rather than
  writing to the real `qmu-co-jp`.
- Keeping the domain layer pure and the LLM provider behind an anti-corruption
  layer, with a deterministic fixture client so CI exercises the whole pipeline
  without credentials or cost.

## 8. Release readiness

Safe to land. CI is defined and green locally; no carry-over concerns block the
release. The only operational follow-up (automated publish stage) is documented
and deferred.
