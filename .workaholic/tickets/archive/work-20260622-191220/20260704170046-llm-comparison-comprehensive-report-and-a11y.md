# Comprehensive LLM comparison (2/2) — full report: methodology, per-aspect analysis, per-model profiles, transparency appendix, and the readability/a11y fix

## Overview

Render the multi-model, multi-trial engine (ticket 1/2) as a **comprehensive,
readable, WCAG-2.2-AA research report** — not a single aggregated table. Consumes
the `ComparisonResult` + per-trial JSON run-artifact from 1/2 and rewrites
`domain/report.ts` to emit a structured page:

1. **Methodology** — models tested (matrix + why the tiers), trial count and how
   aggregates are computed (mean + spread), each probe's definition and grading
   rule, and the exact reproduce commands, credentials, and expected cost.
2. **Per-aspect analysis** — for each probe (speed, max nested-JSON depth,
   length accuracy): a results table showing **mean ± spread (n)** per model, and
   a short factual read of what the numbers show (no marketing language).
3. **Per-model profiles** — a compact per-model block (curated card facts +
   its measured aggregates) so a reader can scan one model end to end.
4. **Data transparency** — the **exact prompt** used for every probe, a compact
   **per-trial summary** (per model×probe: each trial's grade/tokens/elapsed), and
   a link to the committed **JSON run-artifact** holding every trial's verbatim
   raw output. (Owner chose committed JSON artifacts linked from the report.)
5. **Readability / accessibility fix** — resolve the formatting defect the owner
   flagged (**white text on a light/bright background = unreadable**) and guarantee
   the whole page passes `make a11y` (WCAG 2.2 AA): provenance (curated vs measured
   vs fixtured) is signalled by **text/legend and theme-native, AA-contrast
   components**, never hand-rolled low-contrast colored cells.

A **golden-file snapshot** of the fixture-mode report locks formatting so any
regression (including a contrast/structure change) fails CI.

## Considerations

- **Root-cause the contrast defect, don't patch a symptom.** The reported "white text on bright background" comes from colored cells with insufficient contrast. Prefer conveying provenance via **text + theme-native AA components** so the page inherits VitePress's AA-correct theme; if any custom color is unavoidable it must pass pa11y in both themes (`docs/.vitepress/theme/`, `policies/accessibility-first.md`).
- **Page weight / transparency balance.** Raw per-trial *outputs* live in the linked JSON artifact (owner's choice), keeping the page readable; the page still shows exact prompts + a compact per-trial summary. Don't inline megabytes of raw output (`policies/self-explanatory-ui.md`).
- **Publish link resolution.** `make publish` copies only the `.md`; ensure the JSON-artifact link resolves on the corporate Astro site (extend the publish copy set to include the artifact, or link by stable GitHub URL) — coordinate the final path with ticket 1/2 (`scripts/publish-research.sh`).
- **Determinism for the snapshot.** The golden snapshot only holds if fixture mode is byte-stable (guaranteed by 1/2's deterministic fixture); a real `compare` report is expected to differ run-to-run and is **not** snapshotted.
- **Terminology consistency** — the enlarged surface (provider/model/tier/probe/trial/aggregate) must read identically in code, report, sidebar, and diagram (`policies/terminology.md`).
- **Mermaid accessibility** — the Method diagram needs a text alternative/caption to stay AA (`policies/diagram-generation.md`, `policies/accessibility-first.md`).

### Discovered Insights

- **Insight**: Adding the two report pages to the `.pa11yci` URL list surfaced
  **real, pre-existing** WCAG-AA failures that CI never checked — VitePress's default
  Shiki code-token colours (comment `#6A737D` at 4.46:1, keyword `#D73A49` at 4.24:1,
  both just under 4.5:1) in shell code blocks.
  **Context**: The owner's reported "white text on a bright background" was in an
  ad-hoc PDF, but the code-block contrast was a genuine gap; the fix (switch to the
  `github-*-high-contrast` code theme in `docs/.vitepress/config.ts`) is global and
  also fixes the `llm-benchmark` page. The report pages are now in CI's a11y coverage.
- **Insight**: Conveying provenance with **text + labels instead of colour** means
  the report carries zero custom CSS and inherits VitePress's AA-correct theme.
  **Context**: This is the most robust a11y posture — there is no custom contrast to
  get wrong — and it reads identically at any contrast setting or colour-blindness.
- **Insight**: `pa11y-ci` is declared in `docs/package.json` but was not materialised
  in `docs/node_modules`, and the sandbox could not reliably run the preview-server +
  Chromium (errexit + background-process quirks).
  **Context**: The a11y fix was therefore confirmed by HTML inspection of the built
  pages (the failing colours are gone, replaced by ≥4.5:1 values) plus a partial pa11y
  run that had already isolated those as the only violations — the same
  HTML-inspection practice the prior trip used. CI runs the full pa11y gate.
- **Insight**: The JSON run-artifact is linked by a relative path; `make publish`
  copies only the `.md`, so the transparency link will not resolve on the corporate
  Astro site until the artifact is added to the publish copy set (or linked by a
  stable GitHub URL).
  **Context**: Left as a follow-up (`scripts/publish-research.sh`) — it does not affect
  the VitePress preview, tests, or the a11y gate, but should be closed before the page
  is published.
